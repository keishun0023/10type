## 1. 背景（v1から引き継ぎ＋更新）

### プロダクト概要
- **kokorift（ココリフト）**: 生きづらさを扱うセルフCBTプログラム
- 無料診断（現在36問）で「恐れ4軸 × 防衛3軸」のスコアから内在化タイプ（8種）を判定
- 有料プログラム（3,980円・税込・買い切り）: 診断結果とオンボーディング回答をもとに、配合エンジン（決定論）で30日の骨格を組み、AIが枠内の文面を個別生成
- 生成は2フェーズ: preview（Day1〜8、課金前）/ full（30日、課金後）
- 配合エンジン（lib/blending.ts）は現役。恐れ上位×行動/認知比率でProgramConfigを決定論で生成し、AIはその枠内で文面のみ書く
- 技術: Next.js App Router / Supabase / Stripe Checkout / Claude API

### 現在のファネル（これを置き換える）
```
広告 → LP → 無料診断(36問) → 簡易結果画面(長いスクロール) → オンボ質問(別画面) → 深掘り質問 → building(生成待ち) → ペイウォール(別画面) → Stripe決済 → ダッシュボード
```

### 現ファネルの問題
- 簡易結果画面が長い1枚スクロールで、感情のピーク後に機能カタログに切り替わり温度が落ちる
- 結果画面→CTA→ペイウォールで**体験が分断**される（「あ、ここから売り込みか」と気づかれる）
- 旧訴求（無料診断）で来た興味本位層が多く、課金CV 0
- 新訴求（AIカウンセリング）で入ってきた人が、いきなり36問の性格診断を始めるのは**期待とのギャップ**

### 広告戦略の転換
- 旧訴求:「生きづらさのタイプがわかる無料診断」→ 興味本位クリックが多い
- **新訴求:「その"生きづらさ"、AIカウンセリングで解消」**
- ターゲット: 「カウンセリングを検討したことがあるが止まっている人」（25〜35歳中心、慢性的な生きづらさ、料金・予約・対人ハードルで踏み切れていない層）
- CTA: 「まずは無料で体験」or「あなたに合うか確かめる」

### 核となるポジショニング（カウンセリング再フレーム）
- カウンセリングの対話の役割は「認知のクセに気づくこと」。変化を作るのはセッション間の日々の実践（CBTホームワーク）であることが研究で示されている
- このプログラムは「気づき（認知ワーク）」と「実践（行動ワーク）」の両方を毎日届ける
- 「カウンセリングの劣化版」ではなく「カウンセリングが効く理由の部分を、毎日・自分のペースで・1/3以下の価格で」という提案

## 2. 新ファネル：ステップ式1本フロー

### コンセプト
**結果画面とペイウォールの境目をなくす。** 全体を「1画面ずつ"次へ"で進むステップ式フロー」にし、質問→結果→方向性→サービス説明→プラン選択を**1本の連続した体験**として流す。ユーザーからは「売りつけられた」ではなく「導かれた」と感じさせる。

ステップ式が新訴求と合う理由：カウンセリングは一問一答で進む対話形式であり、長いページを読む体験ではない。ステップ式（1画面ずつ進む）の方が「対話されてる感」「自分に向き合ってもらってる感」が出て、「AIカウンセリング体験」の訴求と一致する。

### フロー全体像

```
広告「生きづらさ、AIカウンセリングで解消」
→ LP（FV + CTA「まずは無料で体験」）
→ ステップ式フロー開始:

  [質問フェーズ]
  Step 1〜N:  質問（悩み・場面・困り度・志向 + 診断36問）
              ※「診断してます」ではなく「あなたのことを教えてください」のトーン
              ※ カウンセリングの初回面談（インテーク）の建て付け

  [分析演出]
  Step:       分析中の演出（「あなたの回答を分析しています…」）
              ※ ここでpreview生成をバックグラウンドで開始（実装検討）

  [結果フェーズ]（旧・簡易結果画面に相当）
  Step:       あなたは「◯◯型」です + イラスト + 一言
  Step:       こんなこと、ありませんか?（あるある 2〜3個）
  Step:       どうして、こうなるんだろう?（恐れの構造）
              ※「あなたは悪くない」は独立ステップにせず、ここに一文として溶かす
              （例:「それは性格のせいではなく、〇〇という恐れが行動にブレーキをかけている状態です」）

  [変化のビジョン]（最重要ステップ。AI生成。）
  Step:       こうしたら、こうなります
              ※ 順序: 今のあなた（現状）→ だからこうします（手段）→ そうするとこうなります（未来）
              ※ ユーザーの回答（恐れ上位・志向・difficultScene・自由記述）からAIが個別生成
              ※ 分析演出中にバックグラウンドで生成（preview生成と並行）
              ※ 詳細は§3

  [サービス説明フェーズ]（旧・ペイウォールに相当。境目なく続く）
  Step:       思想（設計の種明かし）
              - 恐れ×防衛の構造が人によって違う
              - だから個別に配合を変える
              - 変わりたい/楽になりたいで認知と行動の比率を変える
              - あなたの30日は、この設計エンジンがあなたの回答から組んだもの
  Step:       具体的な中身
              - 毎日1つ届くワーク（認知/行動の2種類）
              - 1日5〜10分
              - AIがあなた専用に文面を生成
              - ダッシュボードで変化を可視化
  Step:       カウンセリング比較（再フレーム）
              - 週1回の対話で気づく → 毎日の認知ワークで気づく
              - 次のセッションまで放置 → その日のうちに小さく試す
              - 月1〜2万円の継続費用 → 3,980円・買い切り
              - 予約して通って話す → スマホで自分のペースで
  Step:       あなた専用プレビュー（3日分）
              ※ preview生成が完了していればここで表示。未完了なら簡易版orスキップ

  [決済フェーズ]
  Step:       プラン選択 + 価格 + FAQ + Stripe決済CTA

  [課金後]
  → 初回セットアップ（利用条件: 1日に使える時間 / 生活リズム / タイミング）
  → ダッシュボード
```

### 設計上の重要判断

**① 結果とサービス説明の境目を作らない**
ユーザーには「質問→結果→導き」の1本の体験に見える。「ここからが売り込み」と感じさせるCTAボタンや画面遷移を挟まない。「次へ」で自然に流れる。

**② 質問のトーン = カウンセリングのインテーク**
「診断を受けてください」ではなく「あなたのことを教えてください」。36問の性格診断も、カウンセリングの初回聞き取りとして自然に受け入れられるトーンにする。質問UIも、テスト感を出さず対話的に。

**③ 「変化のビジョン」が最重要ステップ**
結果フェーズの終わりに置く。ユーザーの具体的な悩みに対して「今こうです→こうします→こうなります」と因果で描く。**ユーザーの悩みの言葉がそのまま裏返った未来**が一番刺さる（「会議で意見が言えない」→「会議で意見が言えるようになる」）。固定テンプレではなくAI生成にすることで、この精度を出す。
- 「こうするのが良さそうです」（提案・should）→ NG。ユーザーの頭が「なぜ?」に向く
- 「こうしたら、こうなっていきます」（ビジョン・will）→ OK。ユーザーの頭が「そうなりたい」に向く

**④ 「あなたは悪くない」は独立ステップにしない**
スクロール型では感情導線の一部だったが、ステップ式だと1画面まるごと「あなたは悪くない」だけになり説教くさい。「なぜ、こうなるんだろう?」の中に一文として溶かす。

**⑤ CTAに価格を含めない（プラン選択ステップまで出さない）**
価値の説明（思想・中身・カウンセリング比較）を全部見せた後に、初めて価格を出す。文脈なしの3,980円にしない。

## 3. 変化のビジョン：AI生成方式

### 方針
固定テンプレではなく、**ユーザーの回答（恐れ上位・志向・difficultScene・自由記述）を入力として、AIが個別のビジョンを生成する**。理由: ユーザーが「会議で意見が言えない」と書いたなら、未来は「会議で意見が言える」であるべき。固定テンプレではこの精度が出ない。

生成タイミング: **分析演出中（診断完了後）にバックグラウンドで生成**。短い文章（3段落）なので速い・安い。preview生成と同じタイミングで並行可。

### 出力の構造（4段落・順序が重要）

```
第1段落: 根っこの恐れ（構造の指摘）
  あなたの心の中にある恐れを名指しする。「性格のせいではなく、こういう回路が働いている」。

第2段落: それが生活のあちこちに出ている（具体的な現れ方）
  ユーザーの悩み場面を含めつつ、「それだけじゃなく、これもこれも同じ根っこ」と広げる。
  「根っこは同じです」で束ねる。→「全部この恐れから来てたのか」という腹落ちを作る。

第3段落: だからこうします（手段の宣言）
  根っこに対して何をするか。行動寄り/認知寄りで変わる。簡潔に。

第4段落: そうすると、こう変わっていく（未来のビジョン）
  ★最重要。根っこが緩むと、具体的な困りごとも自然と変わる、という構造で描く。
  ユーザーの悩み場面が「裏返った未来」を含む。太字。
  「消える」ではなく「減る」「手放せる」で留め金を守る。
```

**構造の狙い**: 「具体的な困り(症状)を1個ずつ潰す」ではなく、「根っこの恐れが緩むと、具体的な困りが自然と溶けていく」という見せ方。ユーザーは「ああ、全部この恐れから来てたんだ。根っこをやれば全部変わるんだ」と感じる。これが納得感と「やりたい」を作る。

### プロンプトに渡す入力
- 恐れ上位1軸（評価/制御/不完全/関係）
- 志向（変わりたい/楽になりたい）
- difficultScene（選択肢ラベル）
- difficultFreeText（自由記述、あれば）
- タイプ名

### プロンプトの制約（留め金）
- 断定しない（「〜ようになる」「〜瞬間が増える」。「必ず〜」は使わない）
- 疾患名を出さない
- 効果を保証しない
- 「変わりたい」志向の人には行動実験の手段を、「楽になりたい」志向の人には認知ワークの手段を提示
- 「楽になりたい」の人に行動を強制しない（「行動を変える必要はありません」等）
- ユーザーの悩みの言葉をそのまま使って鏡写しにする（自分の言葉が返ってくる＝「わかってもらえた」）
- 全体で300字〜500字程度（納得感を持って読める長さ。短すぎない）

### 出力例

**例1: 評価の恐れ × 変わりたい × 「会議で意見が言えない」**

> あなたの心の中には、「人から低く見られること」への強い恐れがあります。これは性格の問題ではなく、「失敗したら、能力がないと思われるかもしれない」という予測が、無意識のうちに行動にブレーキをかけている状態です。
>
> 会議で意見が言えないのも、ミスを報告する前に何度も確認してしまうのも、新しいことに手を出せないのも——根っこは同じです。「評価されること」が怖いから、評価される可能性のある行動を、気づかないうちに避けている。
>
> このプログラムでは、その根っこに対して、小さな場面から「思ったより大丈夫だった」を一つずつ集めていきます。評価されても実害のない安全な場面から始めて、30日かけて段階的に。考え方の整理も毎日並行します。
>
> 根っこの恐れが少しずつ緩むと、具体的な場面も自然と変わっていきます。**会議で一言、自分の考えを口にできる。ミスをそのまま報告できる。完璧じゃないまま提出して、それでも何も壊れないと知っている。** 人の目が消えるわけではないけれど、それに動きを止められる時間は、確実に減っていきます。

**例2: 制御の恐れ × 楽になりたい × 「予定が決まらないと不安で何度も確認してしまう」**

> あなたの心の中には、「先が読めないこと・コントロールできないこと」への強い恐れがあります。これは心配性という性格の問題ではなく、「読めない＝危険」という回路が強く働いているために、確認や準備を止められなくなっている状態です。
>
> 予定が決まらないと何度も確認してしまうのも、まだ起きていない最悪のシナリオを繰り返し考えてしまうのも、計画を全部決めないと動けないのも——根っこは同じです。「読めないこと」が怖いから、読めるようにするための行動を、やめられない。
>
> このプログラムでは、心配性を直すのではなく、「心配したことのうち、実際に起きたのはいくつだった?」を毎日答え合わせしていきます。心配との距離の取り方を、少しずつ変えていきます。
>
> その答え合わせが積み重なると、恐れの回路が少しずつ緩んでいきます。**予定が決まっていなくても、確認せずにそのまま過ごせる。心配が浮かんでも、「まあ、前も大丈夫だったし」で手放せる。** 心配する力はそのままだけど、それに振り回されて消耗する時間は、減っていきます。

## 4. 質問の仕分け

### 診断フローに統合する（「あなたのことを教えてください」の一部として）
- 診断36問（既存）
- しんどい場面（difficultScene）
- その詳細（difficultDetail / difficultFreeText）
- つらさレベル（distressLevel）
- 変わりたい / 受け入れたい（changeOrientation）

### 課金後の初回セットアップに移動する
- 1日に使える時間（dailyTime）→ preview生成時はデフォルト値、初回セットアップ後にfull生成へ反映
- 生活リズム（lifestyle）
- ベストタイミング（bestTiming）

設計意図: 診断の文脈で聞かれる質問は「自分を理解してもらうための入力」として自然。利用条件の質問は診断の没入感を壊すので、課金後に「あなたに合わせて調整します」として聞く。

## 5. 計測設計

### イベント発火ポイント（ステップ式に合わせて再設計）

- **Purchase**: Stripe決済完了時

### ファネル計測KPI
```
広告クリック → LP → フロー開始(StartTrial) → 質問完了(CompleteRegistration) → サービス説明到達(ViewContent) → プラン選択(InitiateCheckout) → 決済(Purchase)
```
各段の遷移率を計測し、ボトルネックを特定する。

## 6. 表現上の注意（全ステップ共通）

- 「うつが治る」等の疾患への効果断定はNG（薬機法・Meta審査）
- 「生きづらさ」は抽象概念なので「解消」は可
- 対話型チャットと誤解させない（実態は30日プログラム）。「AIカウンセリング」は広告訴求であり、プロダクト内では「AIが生成するあなた専用のプログラム」等の表現にする
- 断定しない（「変わっていきます」「楽になるかもしれません」）
- 煽らない（「このままではダメ」ではなく「こう変わっていける」）

## 7. 実装上の注意

### 既存ファイル
- 診断: `data/questions.ts`、診断フロー本体
- 簡易結果: `components/ResultScreen.tsx`（現在: タイプ表示→あるある→なぜそうなる→CTA→フッター）
- オンボ/ペイウォール: `app/program/page.tsx`（screen state: onboarding → deepdive → building → pricing）
- 生成: `lib/generatePlan.ts`（preview/full）、`lib/schedule.ts`（dailyTime使用）、`lib/prompt.ts`
- 配合: `lib/blending.ts`（決定論でProgramConfig生成）、`data/program.ts`（8部品データ）
- 計測: `lib/pixel.ts`（両ピクセル発火）
- ペイウォールv2: `app/program/v2/page.tsx`（参考のみ。宣伝くさいトーンに注意）

### アーキテクチャ変更
- **ResultScreen + program/page.tsx を統合して、1つのステップ式フローコンポーネントに再構成**
- 各ステップはstate/URLで管理（戻る・計測に対応）
- preview生成のタイミング: 質問完了（分析演出中）にバックグラウンドで開始。プレビューステップまでに完了していなければ簡易版を表示
- dailyTimeは課金後の初回セットアップで取得 → full生成時に反映。preview時はデフォルト値

### 段階的リリース（推奨）
一度に全部変えると効果測定ができない。以下の順を推奨:
1. **結果ステップの追加**（変化のビジョンステップ + ステップ式UIへの移行）
2. **サービス説明ステップの構築**（旧ペイウォールの内容をステップ化）
3. **質問統合**（オンボ質問→診断フロー統合、旧オンボ画面の廃止）
